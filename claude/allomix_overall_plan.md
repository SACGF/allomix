# Donor Chimerism Tool — Overall Build Plan

This is the master plan for building an in-house NGS chimerism monitoring tool. Each step is designed to be fed into a prompt as a self-contained task.

---

## Step 1: Project Setup and README

**Goal:** Establish project structure, write the top-level README, and define scope.

- Create the project directory structure (e.g. `src/`, `tests/`, `third_party/`, `data/`, `docs/`)
- Write a top-level `README.md` covering: what the tool does, the clinical context (HSCT chimerism monitoring replacing STR), the 76-SNP rhAmpSeq panel, intended workflow (genotype donor+host, then analyse post-HSCT admixtures), sensitivity target (<1%), multi-donor support (up to 3 genomes)
- Define input/output contract: what goes in (VCFs? BAMs? both?), what comes out (% chimerism, confidence intervals, per-marker details, QC)
- Write a `CLAUDE.md` for the tool's own repo with dev conventions, dependencies, testing approach
- Set up `pyproject.toml` / packaging basics (Python, dependencies: cyvcf2 or pysam, numpy, scipy)

## Step 2: BAM vs VCF — Determine Primary Input Format

**Goal:** Decide whether the tool should operate on VCFs, BAMs, or both, with clear reasoning.

- Examine the existing VCF files on TAU (`/tau/data/clinical_hg38/idt_rhampseq_sid/`) — what fields are present? Is AF per-sample or INFO-level? Are allele depths (AD) available? Is genotype quality (GQ) present?
- Determine whether the VCF AF tag provides sufficient information for chimerism calculation, or whether we need raw read-level data from BAMs (e.g. for strand bias, per-read allele counting, UMI deduplication)
- Review how Demixtify and EuroForMix handle input — Demixtify works on BAMs, EuroForMix on allele count CSVs. What information do they extract that we'd lose working from VCFs alone?
- Consider the GMP workflow: VCFs are already generated by the existing GATK pipeline. Working from VCFs means zero additional variant calling. Working from BAMs means we'd need to extract allele counts ourselves but get more flexibility.
- Assess whether the GATK per-sample calling accurately represents allele counts at low fractions (e.g. 1% donor = ~0.5% ALT VAF for a het site). Is there a risk of GATK filtering out low-VAF variants as noise in the mixed samples?
- **Decision output:** Primary input format, with fallback strategy. E.g. "VCF-first with BAM fallback for edge cases" or "BAM-only because GATK filtering is a problem for low-fraction mixtures".

## Step 3: Test Data Generation — Synthetic Chimeric Files

**Goal:** Create or find tools to generate synthetic chimeric VCFs (and/or BAMs) at known mixture ratios for validation.

- Search for existing tools that can generate synthetic chimeric data:
  - **BAMSurgeon** — can it create chimeric BAMs by mixing reads from two samples? Can those be converted to chimeric VCFs?
  - **Any VCF blending tools?** — tools that take two VCFs and produce a mixed VCF at a specified ratio
  - Check whether Demixtify, EuroForMix, or chimerism_smmip ship with simulation/test-data-generation scripts
- If no suitable tool exists, write a VCF blending script:
  - Input: donor VCF, host VCF, mixture fraction f, target depth D
  - At each informative marker: calculate expected VAF from donor/host genotypes and f, sample allele counts from Binomial(D, expected_VAF)
  - Output: synthetic mixed VCF with realistic allele counts
- If BAM-level simulation is needed, evaluate BAMSurgeon's `addsnv.py` / read-mixing capabilities
- Generate a test matrix: 13 mixture ratios (0%, 0.1%, 0.5%, 1%, 2%, 5%, 10%, 20%, 50%, 80%, 95%, 99%, 100%) x multiple donor-host pairs
- Include edge cases: related donor pairs (fewer informative markers), samples with low depth at some markers, samples with strand bias
- Write the test data generator as a reusable script in the project

## Step 4: Clone and Examine Reference Open-Source Projects

**Goal:** Clone relevant open-source repos into `third_party/` and study their implementations to inform our design.

### Repos to clone and examine:

**Core references (chimerism-adjacent):**
- `mvynck/Chimerism-Bias` — R code for bias correction (Vynck et al.). Study the correction methodology.
- `mvynck/Chimerism-FABCASE` — R/Shiny app for panel sufficiency evaluation. Extract the allele frequency model.
- `mvynck/Chimerism-nMarkers` — Marker informativity calculations.
- `salipante/chimerism_smmip` — Perl pipeline for smMIP chimerism. Study the multi-donor deconvolution logic (supports up to 4 donors).

**Forensic mixture tools (MLE framework):**
- `Ahhgust/Demixtify` — MLE mixture fraction from biallelic SNPs. Study the likelihood function, error model, and how it handles targeted panels. This is the closest mathematical match to our problem.
- `oyvble/euroformix` — Full probabilistic mixture interpretation. Study the read-depth model for SNP mixtures. More complex than we need but statistically rigorous.

**Contamination/purity tools (simpler approaches):**
- `KhiabanianLab/All-FIT` — Small Python tool, iterative weighted least squares for mixture fraction from VAF. Good reference for a simple implementation.
- `nygenome/Conpair` — Contamination estimation for paired samples. Study paired-sample model.

**Fingerprinting (genotype handling):**
- `brentp/somalier` — Fast genotype extraction and relatedness. Study how it handles genotype extraction at known sites from BAM/VCF.

### For each repo, examine:
- How they model the mixture likelihood (the core math)
- How they handle allele count extraction from VCF/BAM
- How they deal with sequencing error and bias
- How they handle edge cases (no informative markers, extreme ratios, low depth)
- Code quality, test coverage, and whether any code can be directly reused vs. reimplemented
- License compatibility

## Step 5: Detailed Implementation Plan

**Goal:** Synthesize everything learned from Steps 1-4 into a detailed, file-level implementation plan.

- Based on Step 2 (BAM vs VCF decision), define the exact input parsing strategy
- Based on Step 3 (test data), define the validation approach and expected results
- Based on Step 4 (reference code examination), decide:
  - Which mathematical approach to use for chimerism MLE (Demixtify-style? EuroForMix-style? simpler weighted average with bias correction?)
  - Which code (if any) to directly reuse vs. reimplement
  - How to structure the bias correction (Vynck et al. methods)
  - How to handle multi-donor (3-genome) deconvolution
- Write out the implementation plan with:
  - Module-by-module breakdown (genotype comparison, chimerism calculation, bias correction, QC, reporting, CLI)
  - Data flow diagram
  - API/interface design for each module
  - Test plan for each module
  - VariantGrid integration points
  - Dependency list and packaging strategy
